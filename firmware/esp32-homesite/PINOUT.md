# ESP32 Pinout — HomeSite Boiler Unit

## Датчики

| GPIO | Функция | Подключение | Примечание |
|------|---------|-------------|------------|
| 4 | OneWire (DS18B20) | DATA → GPIO4, внешняя подтяжка 4.7kΩ к 3.3V | Все температурные датчики на одной шине |
| 15 | DHT22 | DATA → GPIO15, внешняя подтяжка 10kΩ к 3.3V | Strapping pin — подтяжка обязательна и при загрузке |
| 34 | ADC — давление отопления | Сигнал YD4060 напрямую на пин | ADC1, input-only, по умолчанию |
| 35 | ADC — давление водоснабжения | Сигнал YD4060 напрямую на пин | ADC1, input-only, по умолчанию |

### OneWire шина (DS18B20)

**Подключение каждого датчика — 3 провода** (Vcc/GND/DATA). Парасит-режим (2 провода) официально
поддерживается библиотекой, но на шинах >3 м работает нестабильно — **не использовать**.

- **Vcc (красный) → 3.3V** общий с ESP32
- **GND (чёрный) → GND** общий с ESP32
- **DATA (жёлтый) → GPIO4**
- **Подтяжка 4.7kΩ между DATA и 3.3V — одна на всю шину**. Внутреннего pull-up (~45kΩ) недостаточно —
  шина будет нестабильной, особенно на длинной линии.

**Топология шины**:
- **Daisy-chain** (последовательно, «гирлянда») — правильно. Один кабель от ESP32 до последнего датчика,
  отводы минимальной длины.
- **Звезда** (все датчики отдельными лучами к ESP32) — **нельзя**, даёт отражения сигнала.
- **Длина до 10 м** — обычный провод OK. **10–30 м** — витая пара (DATA+GND в одной паре), экран на GND.
  **>30 м** — отдельный сильный pull-up (2.2kΩ), риски растут.
- **Разрешение**: по умолчанию 11 бит (0.125°C, ~375 мс на конверсию). Менять в `sensor_reader.cpp`
  константой `DS_RESOLUTION` (9–12). 12 бит блокирует цикл на ~750 мс.
- **Адреса сканируются через WiFi-портал или MQTT-команду `scan_sensors`**. Если датчик в логах
  выдаёт `85.0°C power-on reset` — проверить питание и длину/качество шины.

### DHT22 (GPIO15 — strapping pin!)

GPIO15 — bootstrap pin ESP32: **при загрузке должен быть HIGH**, иначе чип не стартует. Подтяжка 10kΩ
к 3.3V, которая требуется для DHT22, одновременно удерживает HIGH при boot — это безопасно.
Но **если забыть подтяжку, ESP32 не загрузится** (будет в download-mode или вечном reset).

### Датчики давления YD WZYDDZ YD4060 (по умолчанию)

Керамический датчик, питание **5V**, линейный выход **0–3.3V**, диапазон **0–0.5 МПа = 0–5 бар**.
Так как выход уже 0–3.3V, делитель напряжения **НЕ нужен** — сигнал идёт прямо на GPIO 34/35.

```
YD4060 (3 провода):
  красный  → +5V (с платы ESP32 / БП)
  чёрный   → GND (общая земля с ESP32)
  жёлтый   → GPIO 34 (отопление) / GPIO 35 (вода)
              │
              ├── 10kΩ ── GPIO (последовательно, опционально — ограничивает ток при сбое)
              │
              └── 100nF ── GND (RC-фильтр против шума ADC)
```

**Защита ADC**: пин ESP32 не толерантен к >3.3V. Если есть риск перенапряжения
(скачок питания, замена на вариант 0–5V), поставить TVS/зенер 3.3V на сигнал к земле.

**Калибровка по умолчанию** (`pressure_reader.h`): zeroV=0.0, maxV=3.3, maxBar=5.0.
Для точности (ADC ESP32 нелинеен на краях) выполнить двухточечную калибровку
по поверенному манометру и при необходимости подкорректировать значения.

### Чек-лист монтажа YD4060

1. **Снять делители напряжения** на плате (4.7kΩ + 3.3kΩ к GPIO 34/35), если они уже распаяны под старый 0.5–4.5V датчик. Иначе сигнал ослабнет до ~41% и давление будет сильно занижено. Можно зашунтировать нижний резистор перемычкой.
2. **Питание датчика — 5V**, не 3.3V. Уточнить по этикетке: бывают версии YD4060 на 9–36V (тогда 5V не запустит датчик), и реже — на 3.3V.
3. **Общая земля** ESP32 и датчика обязательна — иначе ADC прочитает мусор.
4. После прошивки — **двухточечная калибровка по поверенному манометру** (ADC ESP32 нелинеен на 0–0.15V и >2.5V). Записать `analogRead` при 0 бар и при рабочем давлении, при необходимости подкорректировать `_heatZeroV/_heatMaxV` (или `_waterZeroV/_waterMaxV`) в `pressure_reader.h`.

### Подключение другого датчика (например, промышленный 0.5–4.5V)

Нужен делитель (4.7kΩ + 3.3kΩ) → ~0.21–1.85V на вход ESP32:

```
Датчик OUT ──┬── 4.7kΩ ── GND
             │
             └── 3.3kΩ ── GPIO 34/35
```

И вызвать в `setup()` после `pressure.begin(...)`:
```cpp
pressure.calibrateHeating(0.205, 4.0, 1.845);  // zeroV, maxBar, maxV (после делителя)
pressure.calibrateWater(0.205, 6.0, 1.845);
```

## Калибровка и проверка датчиков

> **Per-sensor offset (UI).** Для каждого датчика и каждого его типа данных (tmp/hmt/prs)
> в веб-админке и мобильном приложении есть поле «Оффсеты» (Settings → Оборудование → кнопка
> «Оффсеты» рядом с датчиком). Значение прибавляется к сырым показаниям перед публикацией.
> Backend сохраняет оффсеты в таблице `sensor_offsets`, Gateway отправляет их по MQTT
> командой `{"sensor_offset": "<sensor_name>:<code>:<value>"}`, прошивка применяет к
> `SensorMapping.offsetTmp/offsetHmt` и `PressureReader::setHeatingOffset/setWaterOffset`.
> Оффсеты переживают перезагрузку ESP32 (хранятся в NVS). Чтобы отправить оффсет на
> устройство, у датчика в админке должен быть назначен «Исполнитель» (ESP32, читающий его).

### DS18B20 (температуры)

Датчики заводски калиброваны **±0.5 °C** — для задач отопления/ГВС этого достаточно.
Периодически нужно проверять дрейф.

**Процедура проверки** (раз в год или при подозрении на ошибку):

1. **Эталон — тающий лёд в воде**. Заполнить термос/стакан измельчённым льдом, залить
   чистой водой, перемешать. После 2–3 минут равновесия температура стабильно **0.0 °C**.
2. Опустить датчик в смесь на 1 минуту (не касаясь дна и стенок), прочитать значение
   через MQTT (`home/devices/{node}/sensors/{name}`) или в Serial-логе.
3. Допустимое отклонение: **±0.5 °C**. Если больше — см. ниже.

**Вторая точка (опционально)** — кипящая вода: 100.0 °C при давлении 1 атм (на высоте уровня
моря). В реальности зависит от высоты, некоррекция некритична в пределах 0.1–0.3 °C.

**Если дрейф >0.5 °C**:
- Датчики DS18B20 стоят ~100₽ — проще **заменить**, чем корректировать.
- Если замена невозможна: **выставить offset** в админке (Оборудование → датчик → «Оффсеты» → `tmp`)
  на значение `-error`, где `error = P_fw - P_ref`.

**Типичные причины ложного дрейфа** (до замены проверить):
- плохой контакт на клеммах шины → читается мусор / 85.0 °C;
- длинная звезда вместо daisy-chain → случайные скачки;
- питание датчика просаженное (<3.0 V) → занижение на 1–3 °C;
- самонагрев датчика в плотном кармане гильзы без теплопроводной пасты.

### DHT22 (температура + влажность)

Заводская калибровка **±0.5 °C / ±2–5 %RH**.

**Проверка температуры** — тающий лёд, как для DS18B20. Допуск ±0.5 °C.
**Проверка влажности** — соляной тест: насыщенный раствор NaCl в закрытой банке даёт **75 %RH**
при 25 °C через 8–12 часов. Допуск ±3 %RH.

Для коррекции — выставить оффсет в админке: отдельно для `tmp` (°C) и `hmt` (%RH).
DHT22 деградирует от пыли и химии за 2–3 года — оффсет помогает на время до замены,
при систематическом дрейфе >5 %RH проще поменять датчик (~300₽).

### Датчики давления YD4060

Двухточечная калибровка по поверенному манометру. Нужна один раз при установке и при замене датчика.

> **Быстрый способ** (хватает для бытового котла): если прошивка показывает давление с постоянным
> сдвигом, выставить оффсет в админке (датчик давления → «Оффсеты» → `prs`, в барах). Это не
> исправит нелинейность на краях ADC, но устранит типичный сдвиг нуля. Полная двухточечная
> процедура ниже нужна, только если важна точность на всём диапазоне.

**Шаг 1 — нулевая точка** (датчик снят с магистрали, атмосферное давление):

1. Открутить датчик, дать ему остыть до комнатной температуры.
2. Включить ESP32, дождаться публикации `prs=...` в MQTT (топик `home/devices/{node}/sensors/{prs_heating_name}/prs`).
3. Записать это значение как **текущее давление при 0 бар** — назовём `P0` (идеально 0.00, реально 0.02–0.10).
4. Заодно из Serial-лога ESP32 записать `analogRead` raw-значение. Напряжение: `V0 = raw / 4095 * 3.3`.

**Шаг 2 — рабочая точка** (датчик в системе под давлением):

1. Вкрутить датчик в магистраль, открыть кран.
2. По поверенному манометру зафиксировать реальное давление `P_ref` (бар).
3. Из MQTT считать значение прошивки `P_fw` (бар).
4. Ошибка: `error = P_fw - P_ref`.

**Шаг 3 — правка констант в `pressure_reader.h`**:

Для отопления редактировать строки 38–40, для воды 42–44:

```cpp
// было
float _heatZeroV = 0.0;
float _heatMaxBar = 5.0;
float _heatMaxV = 3.3;
```

- Если нулевая точка даёт `P0` ≠ 0 → сдвинуть `_heatZeroV`:
  `_heatZeroV = V0` (напряжение при 0 бар из шага 1).
- Если рабочая точка даёт ошибку в масштабе → подправить `_heatMaxV`:
  `_heatMaxV_new = _heatZeroV + (V_ref - _heatZeroV) * P_ref / P_fw`,
  где `V_ref` — напряжение в рабочей точке.
- Перепрошить и повторить шаг 2 для контроля.

**Проще практически**: если система стабильна при 1.5 бар, а прошивка показывает 1.65 — умножить
`_heatMaxBar` на `1.5 / 1.65 ≈ 0.909` (было 5.0 → стало 4.55), перепрошить, проверить. Формально
это корректировка масштаба без учёта нелинейности ADC — ±5% точности достаточно для котла.

**Runtime-оффсет** (сдвиг нуля, бар) настраивается через админку — хранится в NVS ключами
`prs_h_off`/`prs_w_off` и применяется в `PressureReader::readHeatingPressure/readWaterPressure`.
Полный пересчёт `zeroV/maxV/maxBar` — только через правку `.h` и перепрошивку.

**Важно**: после замены датчика YD4060 на другой экземпляр — калибровку повторить (разброс между
экземплярами до ±3%).

## Системные

| GPIO | Функция | Примечание |
|------|---------|------------|
| 0 | Кнопка сброса | BOOT кнопка на плате. Удержание 5с = сброс конфигурации |
| 2 | Встроенный LED | Горит = рабочий режим, мигает = чтение датчиков |

## Свободные пины

Следующие GPIO не используются и доступны для расширения:

| GPIO | Примечание |
|------|------------|
| 36 (VP) | ADC1, input-only |
| 39 (VN) | ADC1, input-only |
| 15 | Занят DHT22, свободен если DHT не используется |

## Ограничения ESP32

- **GPIO 34, 35, 36, 39** — только вход (input-only), нет внутренней подтяжки
- **GPIO 6–11** — заняты SPI flash, не использовать
- **GPIO 12** — влияет на напряжение flash при загрузке (bootstrap pin), осторожно при отладке
- **ADC2 (GPIO 0, 2, 4, 12–15, 25–27)** — не работает при активном WiFi, поэтому давление на ADC1 (34, 35)
